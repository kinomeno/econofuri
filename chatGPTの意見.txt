仕様書、かなり整理されています。
特に「ただのネタアプリ」で終わらず、

* ブラウザ完結
* サーバー不要
* note販売との接続
* 無料／有料境界
* 擬態UI
* 将来の常駐化

まで一本の思想で繋がっているので、実装者視点でも理解しやすいです。


実際、Claude Code に渡して作れるレベルまでかなり降りています。
ただし、「このまま実装開始すると詰まりやすい箇所」がいくつかあります。

---

# 強い点

## 1. 「体験」が明確

この仕様書は技術ではなく、

> 「新品紙なのに裏紙っぽく見える」

という一発ネタ体験が中心にある。

しかも、

* 擬態UI
* 緊急Esc
* 「安全管理印刷」
* 「エコだけ大きいロゴ」

まで全部同じ方向を向いています。
これは強いです。
機能追加しても軸がブレにくい。

---

# かなり良い判断

## 「サーバー不要」

これは正解です。

このアプリ、

* PDF送信
* 個人情報
* 社内書類

が絡むので、

> 「ファイルは外部送信されません」

が非常に効きます。


さらに、

* Vercel無料
* GitHub Pages
* Cloudflare Pages

でほぼ維持費ゼロ。

SNS拡散型ツールとして相性が良い。

---

# 実装上、最重要になる箇所

## 「表裏交互PDF」

ここが核です。


ただ、ここは単純に見えて実は面倒です。

理由：

| 入力  | 難しさ |
| --- | --- |
| 画像  | 簡単  |
| txt | 簡単  |
| PDF | 難しい |

PDFは、

* サイズがページごとに違う
* 回転情報
* A4以外
* 埋め込みフォント
* 異常PDF

がある。

なので Claude Code に最初から、

> 「PDFはページ画像化して扱ってよい」

と許容を書いておくとかなり安定します。

つまり：

```txt
PDFはpdf.jsでレンダリングし、
Canvas画像としてpdf-libへ再配置してよい。
完全なPDF構造維持は不要。
```

これだけで難易度がかなり下がる。

---

# 一番重要な技術判断

## 「裏紙ダミーはCanvas描画」

これは非常に良い。


理由：

* 軽い
* 無限生成できる
* 著作権問題が少ない
* 実データ混入を防げる

しかもこのアプリ、

「内容が読めそうで読めない」

くらいがちょうど良い。

なのでむしろ、

* 擬似日本語
* 架空数値
* 意味ありそうな表

を大量生成する方向が向いています。

---

# ここは追加した方がいい

## ダミー濃度スライダー

かなり重要。

例えば：

| モード | 内容            |
| --- | ------------- |
| 薄い  | ほぼ白紙、うっすら罫線   |
| 普通  | 一般的な裏紙        |
| 濃い  | 会議資料っぽい       |
| 狂気  | グラフ・スタンプ・表まみれ |

これだけでかなり遊べる。

---

# 最大の実装リスク

## 「全て印刷」

ブラウザ制限です。


Chrome系は、

* 自動印刷
* 連続ダイアログ
* 両面指定

にかなり制約があります。

なので現実的には：

## 現実ライン

* PDF生成
* blob URL生成
* 新規タブで開く
* `window.print()`

ここまで。

# 逆に、かなり現実的な箇所

## 擬態UI

これはCSSだけでかなり作れます。


特に、

* スプレッドシート風
* タスク管理風

は実装コストに対して満足度が高い。

# かなり重要な追加提案

## 「印刷履歴風ログ」

例えば：

```txt
[完了]
見積_042.pdf
A4 / 両面 / eco mode

[完了]
会議資料.pdf
A4 / recycle safe
```

これを擬態UI内に流す。

かなり雰囲気が出る。

# 実装難易度

## Lv1

実際かなり可能です。

ただし、

* UIを凝りすぎない
* PDF完全再現を捨てる
* 印刷制御を欲張らない

この3つが重要。

---

# Claude Code に追加すると良い一文

かなり効きます。

```txt
重要：
まずは「確実に動く最小構成」を優先してください。
PDF完全再現や高度な印刷制御より、
「ファイル投入 → 表裏PDF生成 → 印刷できる」
体験を優先します。

UIは後から増築しやすい構造にしてください。
```

これで暴走実装を防ぎやすい。
